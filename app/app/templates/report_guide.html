Report Guide

This page is the single source of documentation for PCAP Mapper reports. The Reports page is intentionally a clean launcher; use this guide to decide which export to generate, which files can be imported into MITRE ATT&CK Navigator, and which files are internal PCAP Mapper data.

Recommended report workflow

1. Finish analysis. Upload PCAPs or compressed capture archives and wait for upload, decompression, parsing, and event normalization to complete.

2. Review MITRE coverage. Check Coverage, Data Sources, and Normalized Events pages to understand observed evidence, theoretical telemetry capability, detectable rule coverage, and validated rule matches.

3. Run Validate Rules. If you need validated ATT&CK coverage, run rule validation before generating reports.

4. Generate or refresh reports. Reports are generated from the saved job result, selected ATT&CK source, analysis mode, normalized events, rule validation results, and environment-scoped coverage model.

5. Choose the right output. Use HTML/Markdown for human review, CSV/NDJSON for data handling, Navigator layers for MITRE ATT&CK Navigator, and Other Reports for diagnostics or integration.

MITRE ATT&CK Navigator compatibility

Only files explicitly labeled Navigator Layer are intended to be imported into MITRE ATT&CK Navigator. These files use the Navigator layer JSON structure and include layer metadata such as domain, ATT&CK version, Navigator version, layer version, gradient/color metadata, and a techniques array.

Do not import Enterprise Coverage JSON or Unified ATT&CK Coverage Model JSON into Navigator. Those files are PCAP Mapper application data, not Navigator layers, and Navigator will show layer-version warnings if you try to upload them.

Navigator layers are useful for visual review in ATT&CK Navigator. Application JSON reports are useful for debugging, integrations, audit evidence, and understanding exactly how PCAP Mapper calculated coverage.

HTML Report

Purpose: Primary human-readable investigation report for browser review.

Contents: Analysis configuration, ATT&CK source and mode, environment-scoped coverage summary, coverage-by-tactic rollups, findings, assets, top communications, normalized event evidence, rule-validation evidence, coverage gaps, telemetry recommendations, and report metadata.

Use when: You need a complete analyst-friendly summary to brief a team, review an analysis, or attach to a case.

Do not use for: Machine ingestion.

Navigator importable: No.

Markdown Report

Purpose: Plain-text version of the main report for tickets, documentation systems, Git repositories, and analyst notes.

Contents: Same core information as the HTML report, formatted as Markdown sections and bullet lists.

Use when: You need a portable report that can be pasted into a ticket, wiki, README, incident note, or version-controlled case file.

Do not use for: Structured analytics.

Navigator importable: No.

Hosts CSV

Purpose: Spreadsheet-friendly asset inventory discovered from the analyzed PCAPs and normalized telemetry.

Contents: Host IPs, MAC addresses when available, inferred roles, byte counts, protocols, and first/last seen timestamps.

Use when: You need to review assets, sort by activity, compare hosts, or import asset evidence into another workflow.

Do not use for: Full event evidence.

Navigator importable: No.

Flows CSV

Purpose: Spreadsheet-friendly communication summary.

Contents: Source IP, destination IP, protocol, destination port, packet counts, byte counts, and first/last seen timestamps.

Use when: You need to investigate top talkers, find large transfers, pivot into communications, or validate network-based evidence.

Do not use for: Endpoint-event review or rule diagnostics.

Navigator importable: No.

Events CSV

Purpose: Spreadsheet-friendly normalized event export.

Contents: Parsed event timestamp, platform, log source, event type, host, user, process, source/destination IPs, ATT&CK candidates, confidence, and raw payload excerpt.

Use when: Forwarded logs were present in the PCAP and you need to inspect Windows, Linux, macOS, network, or cloud telemetry recovered from capture payloads.

Navigator importable: No.

Elastic NDJSON

Purpose: Line-delimited JSON export for downstream search/analytics pipelines.

Contents: Hosts, flows, findings, log sources, techniques, and normalized events in index/action pairs suitable for Elastic-style bulk ingestion workflows.

Use when: You want to push PCAP Mapper evidence into a search platform, SIEM lab, or offline analytics process.

Navigator importable: No.

Observed Navigator Layer

Purpose: ATT&CK Navigator layer showing techniques directly evidenced by PCAP traffic and normalized events.

Contents: Navigator-compatible technique entries, scores/colors, comments, and layer metadata for the observed portion of the analysis.

Use when: You want to open the directly observed ATT&CK evidence in MITRE ATT&CK Navigator.

Important: Observed coverage should be conservative. It should represent what the analyzed PCAPs and parsed events actually support, not everything the environment might detect.

Navigator importable: Yes.

Unified Coverage Navigator Layer

Purpose: ATT&CK Navigator layer showing the environment-scoped unified coverage state for each applicable technique.

Contents: Observed, theoretical, detectable, partial, not-covered, and not-applicable states represented with Navigator-compatible colors and comments. Validation status is included as metadata/evidence rather than as a separate overlay color.

Use when: You want a single Navigator view that aligns with the Environment-Scoped Coverage Overlay in the application.

Important: Gray/not-applicable techniques are outside the detected OS/log-source scope and are excluded from coverage denominators.

Navigator importable: Yes.

Theoretical Coverage Navigator Layer

Purpose: ATT&CK Navigator layer focused on what the enterprise should be able to detect based on detected log sources mapped through ATT&CK data-source metadata.

Contents: Detected-log-source telemetry coverage, ATT&CK data-source-derived technique coverage, not-applicable handling, and Navigator metadata.

Use when: You want to review detection capability rather than confirmed activity.

Important: This layer is not proof that an attack occurred. It describes detection opportunity based on telemetry presence.

Navigator importable: Yes.

Enterprise Coverage JSON

Purpose: PCAP Mapper internal enterprise coverage assessment.

Contents: Overall scoped score, maturity, detected coverage scope, scoped denominator, gaps, and telemetry recommendations based on detected OSes and log sources.

Use when: You need to debug coverage scoring, feed another integration, or audit why the UI recommended certain telemetry improvements.

Navigator importable: No.

Unified ATT&CK Coverage Model JSON

Purpose: Complete PCAP Mapper coverage model for all ATT&CK techniques in the loaded dataset.

Contents: Technique ID, technique name, tactic, applicability, observed status, theoretical status, detectable status, partial/not-covered/not-applicable state, rule presence, validation metadata, evidence, confidence, rollups, and counts.

Use when: You need to inspect exactly why the heat map looks the way it does, compare UI/report behavior, or build an external integration around PCAP Mapper coverage decisions.

Navigator importable: No.

Potential ATT&CK Coverage Assessment

Purpose: Human-readable report explaining potential ATT&CK coverage available from detected telemetry products if their richer event types, protocol logs, and forwarding options are fully enabled.

Contents: Observed telemetry coverage, potential coverage, potential increase, detected log sources, per-source observed components, per-source potential components, sample additional techniques, and configuration recommendations.

Use when: You want to understand how to increase realistic theoretical coverage without falling back to generic operating-system assumptions.

Do not use for: Proof that an attack occurred. Potential coverage describes detection opportunity, not observed adversary behavior.

Navigator importable: No.

Rule Validation JSON

Purpose: Explains which Detection Rules and Network Rules matched evidence.

Contents: Rule-validation status, validated techniques, matching rules, rule evidence, and validation details created by the rule engine.

Use when: You need to understand why a technique is considered validated or troubleshoot why a rule did not validate a technique.

Navigator importable: No.

Event Diagnostics JSON

Purpose: Troubleshooting export for the Event Normalization Engine.

Contents: Normalized event summary, mapped events, and a capped list of unmapped events.

Use when: You need to determine whether forwarded logs were parsed correctly, why expected ATT&CK mappings did not appear, or which parsers need improvement.

Navigator importable: No.
